# 飞书 Bot 架构优化 - 技术设计文档

## Context

### 当前架构状态

飞书群组活跃度监测 Bot 采用**模块化分层架构**，当前代码库约 3000+ 行 Python 代码：

```
应用层: main.py, long_connection_listener.py
    ↓
业务逻辑层: collector.py, calculator.py, message_renderer.py, pin_*.py
    ↓
数据访问层: storage.py, auth.py
    ↓
基础设施层: logger.py, rate_limiter.py, utils.py, health_monitor.py
```

### 当前问题

| 问题类别 | 具体表现 | 影响范围 |
|---------|---------|---------|
| **代码重复** | 文件上传逻辑重复 3 次、用户名获取重复 4 次、Pin 处理重复 3 次 | 维护成本高，Bug 修复需多处同步 |
| **功能重叠** | PinMonitor（30秒轮询）与 DailyPinAuditor（每日09:00）功能重叠 | 可能重复处理，数据一致性风险 |
| **同步阻塞** | 单聊回复在主事件循环同步执行，MCP 调用可能导致较长等待时间 | 用户体验受影响 |
| **线程安全** | `long_connection_listener.py` 使用非线程安全 LRU 缓存，`pending_updates` 未加锁 | 多线程环境下可能数据竞争 |

### 约束条件

**硬约束**（不可违反）：
- API 限流：20次/60秒
- Token 生命周期：7200秒，提前300秒刷新
- 消息采集：单次最多5000条，最多100页
- 批量更新：每3条消息更新一次
- 缓存容量：用户名500条，事件去重1000条

**业务约束**（用户需求）：
- Pin 功能：仅保留每日审计（09:00），移除实时轮询
- 单聊处理：保持同步模式，MCP 超时 30 秒，添加重试机制
- 重构策略：逐步重构，优先高风险部分

**技术约束**：
- Python 3.8+
- 依赖库：lark-oapi 1.5.2, requests, Flask, schedule, Pillow
- 数据存储：飞书多维表格（Bitable）+ 云文档（Docx）
- 部署环境：Docker 容器

### 利益相关者

| 角色 | 关注点 |
|------|-------|
| **最终用户** | 功能无退化，响应速度更快 |
| **运维人员** | 部署简单，监控友好，可回滚 |
| **开发人员** | 代码可维护，易于扩展和测试 |

## Goals / Non-Goals

### Goals

1. **消除代码重复**：将重复的文件上传、用户名获取、Pin 处理逻辑统一到服务层
2. **移除功能重叠**：禁用 PinMonitor，统一使用 DailyPinAuditor
3. **优化单聊处理**：保持同步模式，增加 MCP 超时到 30 秒并添加重试机制
4. **提升线程安全**：替换非线程安全组件，保护共享状态
5. **保持功能稳定**：所有现有功能无回归，用户无感知切换

### Non-Goals

**不在此范围**：
- 重写整个项目或改变核心架构（保持模块化分层）
- 修改活跃度计算逻辑或权重配置
- 更换依赖库或数据存储方案
- 添加新功能（如新标签、新统计指标）
- 性能极致优化（如引入异步框架 asyncio）

## Decisions

### 决策 1: 创建 Services 层

**选择**：在现有架构中插入新的 `services/` 层，位于业务逻辑层和数据访问层之间。

**理由**：
- **最小改动**：不破坏现有分层结构，仅在中间插入服务层
- **职责清晰**：服务层封装复杂的业务逻辑（文件上传、Pin 处理），对外提供简单接口
- **易于测试**：服务层独立，可编写单元测试
- **渐进式迁移**：旧代码可逐步调用新服务，不影响现有功能

**架构变化**：
```
原架构：
应用层 → 业务逻辑层 → 数据访问层

新架构：
应用层 → 业务逻辑层 → services/ → 数据访问层
```

**替代方案未选择**：
- ❌ **Repository 模式**：过度设计，当前项目规模不需要
- ❌ **领域驱动设计**：引入复杂概念，团队学习成本高
- ❌ **微服务拆分**：单机部署场景，微服务反而增加复杂度

### 决策 2: 同步单聊处理

**选择**：保持同步处理模式，直接生成图片并回复用户。

**实现方案**：
```python
# 同步处理流程
def handle_private_message(message):
    if is_document_link(message):
        content = mcp_client.fetch_doc(token)    # 获取内容（30秒超时，带重试）
        image = generator.generate_card(content)  # 生成图片
        send_reply(image)                         # 发送图片
```

**理由**：
- **简单直接**：用户发送消息后直接收到图片，体验更好
- **无需占位图**：不需要"生成中"的中间状态
- **代码简洁**：无需后台线程、线程池管理
- **淡绿色样式保持**：继续使用 `HEADER_BG_COLOR = (220, 235, 232)`

**超时和重试配置**：
- MCP 调用超时：30 秒（从 10 秒增加）
- 失败重试：最多 2 次，每次间隔 2 秒
- 降级策略：超时或失败后返回纯文本卡片

**替代方案未选择**：
- ❌ **两阶段异步处理**：用户体验复杂，需要占位图和后台线程
- ❌ **完全异步化**：需要引入 asyncio，重构成本高
- ❌ **消息队列**：单机部署不需要 MQ 的复杂度

### 决策 3: 逐步重构策略

**选择**：按优先级分 3 个阶段实施，每个阶段独立可验证。

**阶段划分**：

| 阶段 | 内容 | 验证标准 | 风险 |
|------|------|---------|------|
| **阶段 1** | 文件上传服务 | 所有调用点功能无回归 | 🟢 低 |
| **阶段 2** | Pin 服务统一 | 移除 pin_monitor.py，每日审计正常 | 🟡 中 |
| **阶段 3** | 其他优化 | 测试覆盖率提升，线程安全 | 🟢 低 |

**理由**：
- **风险可控**：每阶段独立验证，失败时回滚范围小
- **价值优先**：优先解决高风险问题（文件上传、Pin 处理）
- **并行友好**：不同阶段可由不同人员并行开发

### 决策 4: 服务接口设计

**选择**：服务类使用静态方法 + 全局单例模式。

**示例**：
```python
# services/file_upload_service.py
class FileUploadService:
    @staticmethod
    @with_rate_limit
    def upload_to_drive(file_data, auth_token):
        # 实现逻辑
        pass

# 使用方式
from services.file_upload_service import FileUploadService
FileUploadService.upload_to_drive(data, token)
```

**理由**：
- **简洁性**：无需实例化，使用方便
- **无状态**：文件上传无共享状态，静态方法足够
- **兼容性**：与现有 `@with_rate_limit` 装饰器兼容

**替代方案未选择**：
- ❌ **实例化单例**：增加复杂度，无实际收益
- ❌ **依赖注入**：当前项目规模不需要 DI 容器

### 决策 5: 线程安全策略

**选择**：区分有状态和无状态组件，分别处理。

**策略**：

| 组件类型 | 处理方式 | 示例 |
|---------|---------|------|
| **无状态缓存** | 保持不变（单线程使用） | calculator.py 的缓存 |
| **有状态缓存** | 替换为 ThreadSafeLRUCache | user_name_cache, processed_events |
| **共享字典** | 添加 threading.Lock 保护 | pending_updates |

**理由**：
- **精确保护**：只为真正需要的地方加锁，避免性能损失
- **保持简单**：不需要全面引入线程安全，问题导向

## Risks / Trade-offs

### 风险 1: 重构引入新 Bug

**描述**：代码重构涉及多处修改，可能引入未预期的 Bug。

**缓解措施**：
- ✅ 每个阶段编写单元测试，确保行为不变
- ✅ 保留旧代码作为对照，新代码稳定后再删除
- ✅ 灰度发布：先在测试环境验证，再上生产
- ✅ 监控告警：添加日志指标，异常时及时感知

**回滚策略**：
- Git 分支隔离，每个阶段独立分支
- 功能开关控制（环境变量），可快速切换回旧逻辑

### 风险 2: 单聊体验变化

**描述**：两阶段处理可能让用户感到困惑（先收到占位图，再收到完整图片）。

**缓解措施**：
- ✅ 占位图设计友好，明确提示"图片生成中，稍后补充"
- ✅ 时间间隔控制：优化图片生成，确保 10 秒内完成
- ✅ 失败处理：图片生成失败时，发送文本说明

**权衡**：响应速度提升 > 一次完整响应的体验

### 风险 3: Pin 实时提醒缺失

**描述**：移除 PinMonitor 后，用户无法实时收到新增 Pin 通知，需等待每日汇总。

**缓解措施**：
- ✅ 用户沟通：在群内说明变更，提前告知
- ✅ 保留每日汇总：确保信息不丢失
- ✅ 按需恢复：如果用户强烈反馈，可考虑调整策略

**权衡**：简化系统维护 > 实时提醒的便捷性

### 风险 4: 后台线程资源

**描述**：单聊两阶段处理使用后台线程，高并发时可能创建大量线程。

**缓解措施**：
- ✅ 使用 `daemon=True`，主程序退出时自动清理
- ✅ 线程池限制：如果并发量高，改用 `concurrent.futures.ThreadPoolExecutor`
- ✅ 监控线程数：添加日志记录线程创建/销毁

**权衡**：单聊场景并发量低（< 10/min），线程开销可接受

### 风险 5: 依赖外部 MCP 服务

**描述**：单聊依赖飞书 MCP 服务（mcp.feishu.cn），可能不稳定或被防火墙阻断。

**缓解措施**：
- ✅ 超时控制：MCP 调用设置 10 秒超时（从 20 秒缩短）
- ✅ 降级策略：MCP 失败时返回纯文本卡片
- ✅ 重试机制：失败后重试 1 次，仍失败则降级

**权衡**：当前依赖已存在，只是优化超时策略

## Migration Plan

### 阶段 1: 文件上传服务（1-2天）

**步骤**：
1. 创建 `services/file_upload_service.py`
2. 实现 `FileUploadService.upload_to_drive()` 静态方法
3. 编写单元测试

   - `storage.py` → 调用新服务

   - `pin_daily_audit.py` → 使用 PinService

   - 修改 `pin_scheduler.py`，注释掉 PinMonitor 调度

**回滚**：恢复 PinMonitor，环境变量控制启用旧逻辑

**步骤**：
1. 创建 `services/user_service.py`
2. 优化 `reply_card/mcp_client.py`：

5. 为 `pending_updates` 添加锁保护
6. 添加 API 端点常量到 `config.py`
