# 公告沉淀排障 FAQ 卡片（2026-02-25）

## 卡片 1：改到了错误模块，问题仍未解决

- 问题现象：先改了月度归档读取参数后，公告沉淀报错依旧，用户指出“要改的是公告沉淀模块”。
- 根因说明：首次修复定位到了 `monthly_archiver.py`，但实际报错链路在 `long_connection_listener.py` 的公告归档分支。
- 解决方式：将改动聚焦到公告归档链路，仅处理公告写表相关参数，不再触碰月度归档模块。
- 适用提示：同一项目存在多个 Bitable 调用时，先按“日志位置 + 调用链入口”定位模块，再动代码。

## 卡片 2：`WrongRequestBody` 持续出现

- 问题现象：日志返回 `code=1254001, msg=WrongRequestBody`，公告写入失败。
- 根因说明：请求体与目标表结构/配置存在不一致，导致飞书接口拒绝请求。
- 解决方式：先将公告沉淀字段收敛为最小 5 字段，再通过只读接口核对表 ID 与字段定义，按真实结构校准。
- 适用提示：遇到 `WrongRequestBody` 时，不要只改 payload；应同时核对“表 ID、字段名、字段类型”三件事。

## 卡片 3：字段简化后仍报错

- 问题现象：已经只保留 `消息ID/话题ID/发送者姓名/消息内容/发送时间`，仍失败。
- 根因说明：`ARCHIVE_TABLE_ID` 配置成了 app token（`Gbx...`），不是表 ID（`tbl...`），接口实际打到了错误表标识。
- 解决方式：将 `config/.env` 中 `ARCHIVE_TABLE_ID` 改为公告表真实 `table_id`：`tbluetYHwUas3ayf`，并验证 `/records/search` 返回 `code=0`。
- 适用提示：飞书 Bitable 中 `app_token` 和 `table_id` 都会出现在配置里，命名相似，最容易混用；上线前应做一次只读 API 预检。

## 卡片 4：发送时间字段如何降低不匹配风险

- 问题现象：`发送时间` 字段是最常见类型不匹配来源。
- 根因说明：日期/时间列与文本列在飞书侧接收格式不同，类型不一致会导致请求体校验失败。
- 解决方式：按当前决策固定发送文本时间（`YYYY-MM-DD HH:mm:ss`），不再发送时间戳。
- 适用提示：若业务目标是“先稳定入表”，优先使用与表字段已确认一致的格式；再考虑升级为日期类型。
