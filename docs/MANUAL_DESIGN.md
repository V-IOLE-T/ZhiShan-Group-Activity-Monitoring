# 手册设计方案（用户手册 + 管理者手册）

更新时间：2026-02-14

## 1. 目标与原则

本项目建议拆成两本手册，避免角色混淆：

- `用户使用手册`：只讲“星友该怎么用”“怎么触发功能”“怎么看结果”。
- `管理者使用手册`：只讲“系统怎么配”“怎么管”“出问题怎么排查”。

编写原则：

- 先场景后功能：先讲“我要做什么”，再讲“系统原理”。
- 先步骤后说明：每个功能按“前提 -> 操作 -> 结果 -> 常见问题”写。
- 可执行：每一节要包含明确输入（发什么消息/配什么变量）和输出（看到什么结果）。
- 可维护：每本手册附版本号、更新时间、变更记录和负责人。
- 统一术语：公司名固定“至善”，普通用户固定“星友”。
- 图文预留：先写文字版，在关键步骤加截图占位符（如：`[截图建议-01：xxx]`）。

## 2. 建议结构

### 2.1 用户手册结构

1. 你能用它做什么（能力边界）
2. 5分钟上手（最小可用路径）
3. 群内功能
- 标签归档（`#标签`）
- 活跃度统计（发言、回复、@、点赞）
- 被 Pin（加精）后会发生什么
4. 单聊机器人功能
- 发送飞书文档链接
- 发送普通文本
5. 常见错误与自查
6. 使用规范建议（标签规范、消息格式建议）

### 2.2 管理者手册结构

1. 系统概览（模块与数据流）
2. 上线前准备
- 飞书应用权限
- 事件订阅
- 多维表结构
- 文档 token 配置
3. 部署与启动
4. 功能开关管理
5. 日常运营
- 每日巡检
- 每周周报检查
- 每月归档检查
6. 运维与排障
7. 变更与回滚
8. 安全与权限管理

## 3. 文档写法模板（建议统一）

每个功能章节统一采用：

1. 适用角色
2. 前置条件
3. 操作步骤
4. 预期结果（界面/消息/数据变化）
5. 失败表现与处理
6. 相关配置项（管理员章节）

## 4. 版本管理建议

- 文件头固定字段：`版本号`、`更新时间`、`负责人`、`适用环境`。
- 每次改动记录：改了什么、为什么改、影响范围。
- 手册更新触发条件：
  - 新增/下线功能
  - 变量名或表结构变化
  - 权限模型变化
  - 定时任务时点变化

## 5. 当前项目落地建议

基于当前代码和配置，建议：

- 用户手册优先说明：`#标签归档`、`活跃度统计逻辑`、`单聊文档提取`。
- 管理者手册重点说明：`config/.env` 变量治理、`Bitable` 字段一致性、`/health` 巡检、每周/每月调度验证。
- 只写已启用能力，不写未启用能力说明。
